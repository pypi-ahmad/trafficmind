# Local Development

## Prerequisites

| Requirement | Version | Notes |
|---|---|---|
| Python | 3.12+ (3.13 recommended) | venv at `.venv/` |
| Node.js | 18+ | For frontend only |
| GPU + CUDA | Optional | Required for real-time CV inference |

## Python Dependency Model

**Single source of truth:** `pyproject.toml` at the repo root.

| Layer | Extras | What it covers |
|---|---|---|
| Runtime | _(core deps)_ | FastAPI, SQLAlchemy, httpx, numpy, opencv-python, supervision, … |
| Workflow | `[workflow]` | LangGraph, langchain-openai |
| CV / GPU | `[cv]` | Ultralytics, PyTorch, PaddleOCR, PaddlePaddle |
| Dev / test | `[dev]` | pytest, ruff, pytest-cov |

Lock files (`requirements.lock`, `requirements-dev.lock`) are generated by `uv pip compile` and committed. `requirements.txt` is a thin compatibility shim that points to `requirements.lock`.

## Python Environment

```bash
# create venv (if not already present)
python -m venv .venv

# activate
.venv\Scripts\activate           # Windows
source .venv/bin/activate        # Linux/macOS

# install locked dependencies (dev + workflow profile)
pip install -r requirements-dev.lock
pip install --no-deps -e .

# (optional) install CV extras — requires CUDA-capable PyTorch on Linux;
# CPU-only fallback on Windows since PaddlePaddle has no Windows CUDA wheels
pip install -e ".[cv]"

# regenerate lock files after editing pyproject.toml:
# uv pip compile pyproject.toml -o requirements.lock --python-version 3.13 --no-emit-package trafficmind
# uv pip compile pyproject.toml --extra dev --extra workflow -o requirements-dev.lock --python-version 3.13 --no-emit-package trafficmind
```

## GPU / Device Behavior

Two independent GPU stacks exist in the repo — **PyTorch** (vision/detection) and **PaddlePaddle** (OCR). Each resolves its device at startup with no user action required.

| Service | Framework | Default | How GPU is selected |
|---|---|---|---|
| **Vision** (YOLO) | PyTorch | `VISION_DEVICE=auto` | `torch.cuda.is_available()` → `cuda` or `cpu` |
| **OCR** (PaddleOCR) | PaddlePaddle | `OCR_USE_GPU=true` | `paddle.device.is_compiled_with_cuda()` → `gpu:0` or `cpu` |

### What actually uses GPU on a typical machine

**Linux with NVIDIA GPU + CUDA drivers:**
- Vision → CUDA (PyTorch picks up the GPU automatically).
- OCR → CUDA **only if `paddlepaddle-gpu` is installed** (`pip install paddlepaddle-gpu`). The default `paddlepaddle` package is CPU-only.

**Windows with NVIDIA GPU + CUDA drivers:**
- Vision → CUDA (PyTorch CUDA wheels work on Windows).
- OCR → **CPU**. Upstream PaddlePaddle does not publish Windows CUDA wheels; the engine logs a warning and falls back. This is expected behavior — not a bug.

**No GPU / CI:**
- Both services resolve to CPU silently. All tests (including smoke and integration) pass on CPU.

### Verify on your machine

```bash
# PyTorch GPU
python -c "import torch; print('vision:', 'cuda' if torch.cuda.is_available() else 'cpu')"

# PaddlePaddle GPU
python -c "import paddle; print('ocr:', 'gpu:0' if paddle.device.is_compiled_with_cuda() else 'cpu')"
```

### Force overrides

```bash
# Force vision to CPU even when GPU exists
VISION_DEVICE=cpu

# Force OCR to CPU (skips Paddle auto-detect)
OCR_USE_GPU=false
# — or —
OCR_DEVICE=cpu
```

## Database Setup

### SQLite (Default for Local Development)

No setup required. The database file `trafficmind.db` is created automatically.

```bash
# create tables via Alembic
alembic upgrade head

# or create tables and seed demo data in one step
python -m apps.api.app.demo.seed --create-schema
```

### PostgreSQL

Set `DATABASE_URL` in your `.env`:

```
DATABASE_URL=postgresql+asyncpg://user:password@localhost:5432/trafficmind
```

Then run migrations:

```bash
alembic upgrade head
```

## Running Services

### Backend API

```bash
uvicorn apps.api.app.main:app --reload --port 8000
```

The API serves at `http://localhost:8000`. OpenAPI docs are available at `/api/v1/docs`.

Health probes:

- Liveness: `GET /api/v1/health`
- Readiness: `GET /api/v1/health/ready`

### Workflow Service

```bash
uvicorn apps.workflow.app.main:app --reload --port 8010
```

The workflow service runs with the deterministic `heuristic` provider by default. No API key is required for local execution today.

Health probes:

- Liveness: `GET /api/v1/health`
- Readiness: `GET /api/v1/health/ready`

### Frontend

```bash
cd frontend
npm install
npm run dev
```

Serves at `http://localhost:3000`. Expects the backend API at the URL configured in `TRAFFICMIND_API_BASE_URL` or `NEXT_PUBLIC_TRAFFICMIND_API_BASE_URL`.

## Demo Data

Seed the database with realistic synthetic data for local walkthroughs:

```bash
# create schema + seed (fresh database)
python -m apps.api.app.demo.seed --create-schema

# seed only (tables already exist)
python -m apps.api.app.demo.seed

# list available scenarios
python -m apps.api.app.demo.seed --list-scenarios

# write a JSON walkthrough report
python -m apps.api.app.demo.seed --report-path demo-seed-report.json
```

The seeder is idempotent. It replaces existing demo records on each run.

**What gets seeded:** 3 cameras, 4 streams, 5 zones, 4 detection events, 3 plate reads, 3 violations, 3 evidence manifests, 3 workflows. Cameras are configured for online, degraded, and offline health states.

**After seeding, try:**

- `GET http://localhost:8000/api/v1/cameras` — list demo cameras
- `GET http://localhost:8000/api/v1/observability/dashboard` — health dashboard
- `GET http://localhost:8000/api/v1/plates` — plate reads
- `POST http://localhost:8000/api/v1/exports` — create a case export bundle

## Running Tests

```bash
# full suite
python -m pytest tests/ -q

# specific module
python -m pytest tests/rules/ -v
python -m pytest tests/api/ -v

# with coverage report
python -m pytest tests/ --cov=apps --cov=services --cov-report=term-missing

# skip GPU-dependent integration tests
python -m pytest tests/ -m "not integration"

# repeatable local/CI validation surface
python infra/scripts/run_checks.py --suite all

# golden-path regression (one-command critical-path validation)
python infra/scripts/run_checks.py --suite golden-path
```

### Golden-Path Regression

The `golden-path` suite validates the full critical path in a single command:

1. **Migrate DB** — runs `alembic upgrade head` against a temporary SQLite database to verify the migration chain is intact.
2. **Smoke pipeline** — exercises detection → tracking → OCR → rules → persist end-to-end with stub backends (no GPU, no network).
3. **API + workflow routes** — runs API foundation tests (health, config, route registration) and workflow HTTP tests (execution, interrupt/resume).
4. **Frontend build** — runs `npm run build` in the frontend directory to catch compile errors.

```bash
python infra/scripts/run_checks.py --suite golden-path
```

## Troubleshooting

### `ModuleNotFoundError` on imports

Ensure the package is installed in editable mode:

```bash
pip install -e ".[dev]"
```

### CUDA not detected

```bash
# check PyTorch GPU (vision)
python -c "import torch; print(torch.cuda.is_available())"

# check PaddlePaddle GPU (OCR)
python -c "import paddle; print(paddle.device.is_compiled_with_cuda())"
```

If PyTorch returns `False`, reinstall with CUDA support from [pytorch.org](https://pytorch.org).

If Paddle returns `False` and you need GPU OCR, install the CUDA wheel:
```bash
pip install paddlepaddle-gpu   # Linux only; no Windows CUDA wheel available
```

### Alembic migration conflicts

```bash
alembic history --verbose
alembic upgrade head
```

### Frontend can't reach API

Check `TRAFFICMIND_API_BASE_URL` or `NEXT_PUBLIC_TRAFFICMIND_API_BASE_URL` in `.env` and ensure the backend is running on the expected port.
